采用POM模式

POM的优势?

  • 简单来说就是：

    1. 采用数据与用例分离模式。数据、代码易维护。要是哪个界面的UI改了，那么你就可以只改那个界面的baseView.py就可以了，不需要整个代码模块都改。
    2. 可复用的代码合在一起，使代码更整洁。
    3. 每个UI对应一个BaseView.py。

  • 详细的说就分为几点：

    1. POM提供了一种在UI层操作、业务流程与验证分离的模式，这使得测试代码变得更加清晰和高可读性
    2. 对象库与用例分离，使得我们更好的复用对象，甚至能与不同的工具进行深度结合应用
    3. 可复用的页面方法代码会变得更加优化
    4. 更加有效的命名方式使得我们更加清晰的知道方法所操作的UI元素。例如我们要回到首页，方法名命名为: goHomePage(),通过方法名即可清晰的知道具体的功能实现。

什么是POM（Page Object Model）？

  • 页面对象模型(POM)是一种设计模式，用来管理维护一组web元素集的对象库
  • 在POM下，应用程序的每一个页面都有一个对应的page class
  • 每一个page class维护着该web页的元素集和操作这些元素的方法
  • page class中的方法命名最好根据其对应的业务场景进行，例如通常登录后我们需要等待几秒中，我们可以这样命名该方法: waitingForLoginSuccess().

项目思路

项目设计

  • 一个模块（被测项目的页面）对应一个py文件及一个测试类（测试文件）
  • 每一个测试页面（系统的页面）中存储页面元素及此页面中涉及到的功能
  • 每一个用例组合在一个测试类里面生成一个py文件

项目目标

  • 生成测试用例执行结果报告
  • 生产测试用例执行日志
  • 用例执行失败或执行完成后自动发生邮件报告
  • 用例执行失败或者成功时截取图片
  • 数据驱动（读取测试数据，减少脚本维护成本）

项目目录

app # 存放APP包，如果没有安装，则启动的时候调用该模块安装APP
baseView 

businessView # 每个UI页面对应一个PageView.py
    loginView.py # 登录界面
common
    common_fun.py # 存放公共方法，比如判断是否开启定位等。
    desired_caps.py # 读取APP的配置信息
    myunit.py # 启动APP+关闭APP
config
    caps.yaml # 存放APP配置信息
    log.conf # 日志配置信息
data
    test_data.csv # 存放测试数据
    elementDate.csv # 存放元素信息
logs # 存放用例执行过程中的log信息
reports # 测试报告
screenshots # 截图
    Fail # 存放用例执行失败时截图
    Pass # 存放用例执行成功时截图
test_case # 测试用例，以test_x开头
    test_login.py # 登录测试
test_run # 执行测试用例。要执行哪些测试用例直接在在里面添加就行
